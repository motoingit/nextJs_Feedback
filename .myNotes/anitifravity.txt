# Debugging Logs, Comments, and Error/Blindspot Resolution Plan

This plan aims to add detailed debugging console logs, descriptive comments, and fix critical/fatal design errors and blind spots across the anonymous feedback Next.js project.

## User Review Required

> [!IMPORTANT]
> The following architectural and configuration issues have been identified and will be fixed as part of this plan. Please review them.

1. **Authentication API Routes & Type Mismatches**:
   - In `types/next-auth.d.ts`, `isAcceptingMessage` was typed as `string` in some places and `boolean` in others, creating type-safety issues during NextAuth session resolution. We will unify this to `boolean`.
   - In `options.ts`, the NextAuth credentials provider had fields defined for `email`, but in the authorization flow it accessed `credentials.identifier`. This discrepancy could cause issues when processing requests. We will update the credentials key to `identifier`.
   - The custom sign-in page redirection in `options.ts` was pointing to `/signin`, but the project's folder is `app/(auth)/sign-in/page.tsx` (accessible via `/sign-in`). This would lead to a 404 page when NextAuth redirects users. We will correct the path to `/sign-in`.

2. **Username Uniqueness Route (GET vs. POST)**:
   - In `app/api/check-username-unique/route.ts`, the route was registered as `POST`, but it retrieved search params via `new URL(req.url)`. This is typically used for `GET` requests. We will change the method to `GET` to match its logic.

3. **Sign-up Duplicate Key Errors**:
   - In `app/api/sign-up/route.ts`, a major blind spot existed: if a username was taken by an unverified user, another email registration with that username would crash with a MongoDB E11000 duplicate key error on save, returning a 500 error.
   - Also, when an unverified email tried to sign up again with a new username, the database document kept the old username since it was never updated.
   - We will resolve this by:
     - Checking if the username is taken by any user (verified or unverified).
     - Allowing updating of the username when registering again with the same unverified email.

4. **Zod Validation Error Messages Configuration**:
   - In `schemas/`, Zod validations were using `{ error: '...' }` instead of the correct syntax `{ message: '...' }`. The custom messages were ignored by Zod. We will correct these.
   - `schemas/messageSchema.ts` was exporting `acceptMessageSchema` instead of `messageSchema`, conflicting with `schemas/acceptMessageSchema.ts`. We will rename the export.

## Open Questions

*No critical open questions. We will proceed with implementing the standardized debugging logs and clean comments alongside the bug fixes.*
